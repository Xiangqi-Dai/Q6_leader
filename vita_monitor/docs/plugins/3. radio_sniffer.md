# radio_sniffer 无线信号嗅探

Target：对周边可见的 WiFi 信号进行嗅探，供前端渲染 **2.4 GHz / 5 GHz 各信道的占用情况**。

参考实现：`reference/嗅探/`（Windows `netsh` 原型）；硬件端在 **Linux** 上通过 `iw` / `nmcli` 完成同等采集。

---

## 1. `hardware_sender` 数据采集

- 插件类：`plugins.radio_sniffer.plugin.RadioSnifferVitalPlugin`
- 采集逻辑：`helpers.radio_sniffer.RadioSniffer`
  - **Linux**：优先 `iw dev <iface> scan`（回退 `scan dump`），失败时尝试 `nmcli device wifi rescan` + `device wifi list`
  - 同一 BSSID 多条记录时保留 **RSSI 最强** 的一条
- **异常处理**：单次扫描失败仍上报快照，`status` 为 `empty` 或 `error`，`networks` 为空，`channel_occupancy` 为零占用；不中断插件进程。

### `config.yaml` 的 `kwargs`

| 参数 | 说明 |
| --- | --- |
| `interface` | 无线接口名，如 `wlan0`；留空则自动选择第一个 `iw dev` 或 nmcli WiFi 设备 |
| `scan_method` | `auto`（先 iw 后 nmcli）、`iw`、`nmcli` |

### 运行依赖（Linux）

- 已启用的无线网卡
- 建议安装 `iw`；若使用 NetworkManager，需可用的 `nmcli`
- 部分环境对 `iw scan` 需要 root 或 `CAP_NET_ADMIN`；无权限时可改用 `scan_method: nmcli`

### 建议采集周期

扫描耗时通常数秒，`interval_sec` 建议 **≥ 15**（默认配置 15s）。

## 2. `hardware_sender` → `backend_receiver`（`vita_data`）

```json
{
  "vita_type": "radio_sniffer",
  "vita_data": {
    "status": "ok",
    "interface": "wlan0",
    "scanned_at": 1715302000.123,
    "network_count": 12,
    "networks": [
      {
        "ssid": "Lab-WiFi",
        "bssid": "04:67:61:90:6c:0b",
        "rssi_dbm": -62,
        "signal_percent": 76,
        "channel": 48,
        "band": "5_ghz",
        "authentication": "WPA2",
        "encryption": "CCMP"
      }
    ],
    "channel_occupancy": {
      "2_4_ghz": {
        "channels": [1, 2, 3],
        "ap_counts": [0, 2, 1],
        "peak_rssi_dbm": [-95, -55, -70]
      },
      "5_ghz": {
        "channels": [36, 40, 48],
        "ap_counts": [1, 0, 3],
        "peak_rssi_dbm": [-80, -95, -62]
      }
    },
    "error": ""
  }
}
```

| 字段 | 说明 |
| --- | --- |
| `status` | `ok` 有网络；`empty` 扫描成功但未发现 AP；`error` 命令/权限失败 |
| `networks[]` | 可见 AP 列表，字段与参考 JSON 对齐 |
| `channel_occupancy` | 按频段汇总的信道 AP 数与最强 RSSI，供前端画占用图 |
| `error` | 失败时的简短原因（成功时为空字符串） |

---

## 3. `backend_receiver` 数据处理

- 插件：`plugins.radio_sniffer.plugin.RadioSnifferBackendPlugin`
- **透传**发送端 `vita_data`，不做字段裁剪；由 `publish_vital_update` 推送到 WebSocket。

`backend_receiver/config.yaml` 的 `receiver.plugins.radio_sniffer` 需 `enabled: true`。

---

## 4. `backend_receiver` → `frontend_dashboard`

与发送端 `vita_data` 结构一致（见第 2 节）。

WebSocket 订阅：设备详情页 `subscribe` 需包含 `radio_sniffer`（见 `frontend_dashboard/src/infra/api.ts`）。

---

## 5. `frontend_dashboard` 前端 UI

- **详情图**：`RadioSnifferChannelPanel`（ECharts，双频段堆叠），位于地图与心跳折线图之间
- **实时底栏**：`RadioSnifferChannelStrip`（与心跳 `VitalColumn` 底栏同布局），紧贴心跳标签栏上方
  - 上行：状态 / AP 数 / 接口 / 扫描时间（chip）
  - 下行：**2.4G、5G** 各一行通道色块（绿≤2 · 橙≤4 · 红拥堵），WebSocket 实时刷新
- 渲染逻辑对齐 `docs/plugins/嗅探/visualizer.py` 的 `plot_channel_occupancy`：
  - 通道条颜色按 AP 数量：绿（≤2）→ 橙（≤4）→ 红（更 congested）
  - 散点大小/颜色按 `signal_percent`；悬停显示 SSID、信道、RSSI
- 数据来自 `detail.latest.radio_sniffer` 与 `vital.updated` 实时更新
