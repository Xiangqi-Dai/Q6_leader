# ip_peer_monitor IP 对端连通性监控

监控**机器人本体**与**通过 IP 与机器人联络的各对端设备**之间的网络连通性（局域网或可达的专网 IP），与 `heartbeat`（机器人 ↔ 公网）互补。

| 对比项 | `heartbeat` | `ip_peer_monitor` |
| --- | --- | --- |
| 探测目标 | 公网 DNS（`8.8.8.8` / `1.1.1.1`） | 配置中的对端 IP 列表 |
| 业务含义 | 机器人是否还能上云 | 机器人与周边 IP 设备链路是否健康 |
| 典型场景 | 4G/Wi‑Fi 出网质量 | 激光雷达、底盘、工控机、伴随机等 |
| 采集频率建议 | 2s（较重，含 RSSI 等） | **5s 短测**（轻量 ICMP） |

---

## 0. 方案总览

```text
config.yaml (peers 列表)
        │
        ▼
IpPeerMonitorVitalPlugin.collect_vita_data()
        │
        ├── helpers/network_info/NetworkInfo.probe_peers(...)
        │     └── func/peer_connectivity_collector（短 ICMP，多目标）
        │           └── func/icmp_probe（与公网探测共用解析）
        │
        ▼
vita_data（含 peers[] 逐目标指标）
        │
        ├── MQTT → backend_receiver（透传）
        ├── frontend_dashboard（对端状态表 + 可选时序）
        └── local_dashboard Panel（断网时 LAN 内仍可看）
```

**设计原则**

1. **短测省带宽**：每周期对每个对端仅发少量 ICMP（默认 2 包），`interval_sec` 默认 **5.0**。
2. **配置驱动**：对端列表仅来自本 plugin 的 `kwargs.peers`，不写死在代码里。
3. **失败可观测**：单目标探测失败不阻断整包上报；失败字段使用与 `heartbeat` 一致的断联哨兵值。
4. **Helper 归一**：**不**单独建 `helpers/ip_peer_monitor/`；所有网络连通性探测统一归入 `helpers/network_info/`（公网 + IP 对端），plugin 只编排与组装 `vita_data`。

| 项 | 值 |

Topic 片段：`robot/{device_id}/vitals/ip_peer_monitor`

### 与 `network_info` 的关系

`helpers/network_info/` 是硬件端**所有网络连通性探测**的共享模块，按探测对象分两类 API：

| 探测对象 | `NetworkInfo` 方法 | 底层 Collector | 消费插件 |
| --- | --- | --- | --- |
| 公网（出网质量） | `get_connection_info()` 等 | `connection_quality_collector` | `heartbeat` |
| 配置的对端 IP 设备 | `probe_peers(...)` | `peer_connectivity_collector` | `ip_peer_monitor` |

两类探测共用 `func/command_runner.py` 与 `func/icmp_probe.py`（从现有 ICMP 解析抽取），哨兵值常量亦统一维护。

## 2. `hardware_sender` 数据采集

### 2.1 插件与 `network_info` 职责

- **插件** `IpPeerMonitorVitalPlugin`：读取 `kwargs`，调用 `NetworkInfo.probe_peers(...)`，将返回结构组装为 `vita_data`。
- **`NetworkInfo`**（`helpers/network_info/network_info.py`）新增门面方法：

```python
def probe_peers(
    self,
    peers: list[dict[str, str]],
    *,
    ping_count: int = 2,
    ping_timeout_sec: float = 2.0,
    max_peers_per_cycle: int = 0,
) -> dict[str, Any]:
    """返回含 peers[]、peer_count、online_count、probed_at、error 的探测结果。"""
```

- **`PeerConnectivityCollector`**（`func/peer_connectivity_collector.py`）：
  - 校验 `peers` 配置（合法 IPv4/IPv6 或主机名；非法项跳过并记入汇总 `error`）。
  - 对每个对端执行**短 ICMP**（见 §2.3），内部调用 `IcmpProbe.measure(host, count, timeout)`。
  - 解析延迟、丢包率；推导 `connected`（见 §2.4）。
  - **不**直接发布 MQTT。

**重构（与 `heartbeat` 同步）**：将 `connection_quality_collector.py` 中的 ICMP 命令构造与输出解析抽取为 `func/icmp_probe.py`；`ConnectionQualityCollector` 改为对公网目标（`8.8.8.8` / `1.1.1.1`）调用 `IcmpProbe`，`PeerConnectivityCollector` 对配置 IP 调用同一套解析，避免重复实现。

### 2.2 `config.yaml` 配置

```yaml
sender:
  plugins:
    ip_peer_monitor:
      enabled: true
      class: plugins.ip_peer_monitor.plugin:IpPeerMonitorVitalPlugin
      interval_sec: 5.0          # 监测频率：每 5s 一轮短测
      qos: 0
      kwargs:
        peers:
          - ip: "192.168.1.100"
            name: "激光雷达"
          - ip: "192.168.1.101"
            name: "底盘控制器"
          - ip: "192.168.8.89"
            name: "空气传感器"
        ping_count: 2            # 每目标每轮 ICMP 包数，默认 2（平衡精度与带宽）
        ping_timeout_sec: 2.0      # 单目标 subprocess 超时（秒）
        max_peers_per_cycle: 0   # 0=每轮探测全部；>0 时轮询子集（对端很多时降负载）

local_dashboard:
  panel_order:
    - heartbeat
    - ip_peer_monitor
    # ...
  vitals:
    ip_peer_monitor:
      push_to_ui: true
      ui_history_limit: 60       # 约 5 分钟 @ 5s 间隔
```

#### `kwargs` 字段说明

| 参数 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| `peers` | `list[object]` | **必填** | 对端列表；每项至少含 `ip` |
| `peers[].ip` | `string` | 必填 | 对端 IP 或主机名 |
| `peers[].name` | `string` | 同 `ip` | 展示用别名（前端 / Local Dashboard） |
| `ping_count` | `int` | `2` | 每目标每轮发包数；**勿超过 4**（带宽控制） |
| `ping_timeout_sec` | `float` | `2.0` | 单次 `ping` 命令超时 |
| `max_peers_per_cycle` | `int` | `0` | `0` 表示每轮全量；`N>0` 时每轮只测 `N` 个，按列表顺序轮询，适合 >10 对端 |

#### 带宽与耗时估算

单目标短测（`ping_count=2`）典型耗时 < 500ms（LAN）。设对端数为 `P`：

- 全量模式：`max_peers_per_cycle=0` 时，单轮最坏约 `P × ping_timeout_sec`。
- 若 `P × ping_timeout_sec` 可能超过 `interval_sec`，应调大 `interval_sec` 或启用 `max_peers_per_cycle`。

示例：20 个对端、`ping_timeout_sec=2` → 最坏 40s，**必须**轮询或加大间隔。

### 2.3 短 ICMP 策略

与 `heartbeat` 使用相同跨平台 `ping` 参数风格，但**减少包数**：

| 平台 | 命令形态（示意） |
| --- | --- |
| Linux | `ping -c {ping_count} -W {ceil(ping_timeout_sec)} {ip}` |
| Windows | `ping -4 -n {ping_count} {ip}` |
| macOS | `ping -c {ping_count} -W {ms} {ip}` |

首版采用**串行**探测各对端；后续若有性能需求再评估并行子进程（文档预留，首版不实现）。

### 2.4 指标推导与断联哨兵

与 `docs/plugins/1. heartbeat.md` 对齐：

| 字段 | 成功 | 失败 / 无响应 |
| --- | --- | --- |
| `connected` | `true`（收到 ≥1 个 ICMP 应答且 `packet_loss_rate < 1.0`） | `false` |
| `latency_ms` | RTT 样本均值或摘要 `avg`，**取整** | `999` |
| `packet_loss_rate` | `0.0–1.0`（解析 ping 输出） | `1.0` |
| `error` | `""` | 简短原因，如 `timeout` / `unreachable` / `invalid_host` |

**异常处理**：单个对端探测异常时，该条写入哨兵值，**仍上报整包**；插件进程不退出。

### 2.5 运行依赖

- 系统自带 `ping` 命令。
- 机器人侧需有到对端的 ICMP 权限（部分环境需 `CAP_NET_RAW` 或 root；与 `heartbeat` 相同）。
- 对端若禁 ICMP，会出现 `connected: false`；此类设备后续可扩展 `probe: tcp`（**不在首版范围**）。

## 3. `hardware_sender` → `backend_receiver`（`vita_data`）

```json
{
  "vita_type": "ip_peer_monitor",
  "vita_data": {
    "status": "ok",
    "probed_at": 1715302000.123,
    "peer_count": 3,
    "online_count": 2,
    "peers": [
      {
        "ip": "192.168.1.100",
        "name": "激光雷达",
        "connected": true,
        "latency_ms": 2,
        "packet_loss_rate": 0.0,
        "error": ""
      },
      {
        "ip": "192.168.1.101",
        "name": "底盘控制器",
        "connected": false,
        "latency_ms": 999,
        "packet_loss_rate": 1.0,
        "error": "timeout"
      },
      {
        "ip": "192.168.8.89",
        "name": "空气传感器",
        "connected": true,
        "latency_ms": 5,
        "packet_loss_rate": 0.5,
        "error": ""
      }
    ],
    "error": ""
  }
}
```

### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `status` | `string` | `ok`：至少完成一轮探测；`error`：配置无效或全局探测失败 |
| `probed_at` | `number` | 本轮探测完成时的 Unix 时间戳（秒，可带小数） |
| `peer_count` | `int` | 配置中对端总数（含无效项过滤前或后的数量，实现时固定为**有效**对端数） |
| `online_count` | `int` | `connected=true` 的对端数量 |
| `peers[]` | `array` | 逐对端结果，顺序与配置 `peers` 一致 |
| `peers[].ip` | `string` | 对端 IP |
| `peers[].name` | `string` | 展示名 |
| `peers[].connected` | `bool` | 是否连通 |
| `peers[].latency_ms` | `int` | 往返延迟（ms） |
| `peers[].packet_loss_rate` | `float` | 丢包率 `0.0–1.0` |
| `peers[].error` | `string` | 单目标错误（成功为空串） |
| `error` | `string` | 插件级错误（如 `peers` 为空；成功为空串） |

`peers` 为空时：`status=error`，`peer_count=0`，`online_count=0`，`peers=[]`，`error` 说明原因。

## 4. `backend_receiver` 数据处理

- 插件：`plugins.ip_peer_monitor.plugin.IpPeerMonitorBackendPlugin`
- 策略：**透传**发送端 `vita_data`（与 `radio_sniffer` 相同），可选做轻量校验（`peers` 为 list、`packet_loss_rate` 范围钳制）。
- 历史：默认 `persist_vital_history: true`，写入 ClickHouse 供云端曲线查询（与 `heartbeat` 一致）。

`backend_receiver/config.yaml` 示例：

```yaml
receiver:
  plugins:
    ip_peer_monitor:
      class: "plugins.ip_peer_monitor.plugin:IpPeerMonitorBackendPlugin"
      enabled: true
      kwargs: {}
```

### 6.2 阈值着色

| 指标 | 绿 | 黄 | 红 |
| --- | --- | --- | --- |
| `connected` | `true` | — | `false` |
| `latency_ms` | ≤ 10 | ≤ 50 | > 50（LAN 较 heartbeat 更严） |
| `packet_loss_rate` | < 1% | < 5% | ≥ 5% |

### 6.3 数据来源

- 实时：`vital.updated` → `detail.latest.ip_peer_monitor`

### 6.4 机队总览（可选）

`HomeView` 设备卡片角标：若存在 `online_count < peer_count`，显示黄色警告点（非阻塞上线）。

## 10. 联调检查清单

- [ ] `hardware_sender` / `backend_receiver` 均启用 `ip_peer_monitor`，`vita_type` 一致。
- [ ] `kwargs.peers` 中 IP 在机器人上 `ping` 可达（或确认禁 ping 时的预期离线）。
- [ ] `interval_sec: 5.0`，单轮耗时 < 5s（对端过多时调 `max_peers_per_cycle`）。
- [ ] WebSocket 收到 `vita_type=ip_peer_monitor` 的 `vital.updated`。
- [ ] 断开某一 LAN 设备后，对应 `peers[].connected` 变 `false`，`online_count` 递减。
- [ ] 公网断联 + Local Dashboard 启用时，内网页仍刷新对端表。
