# heartbeat 心跳体征数据

表示  具身智能体 与 公网的连接情况

### `hardware_sender` 数据采集

采集由 `HeartbeatVitalPlugin` 周期性调用 `helpers.network_info.NetworkInfo` 的**公网侧 API**（`get_network_type` / `get_my_ip` / `get_connection_info`）完成；
各子采集器在 `hardware_sender/helpers/network_info/func/` 下实现。对端 IP 设备连通性由同模块的 `probe_peers` 提供，供 `ip_peer_monitor` 使用（见 `docs/plugins/5. ip_peer_monitor.md`）。

- 异常处理：单次采集异常时**不中断上报**： 将采集失败的字段赋值为能表示断联的值。

- **网络连接方式 `network_type`**：`NetworkTypeCollector` 根据**默认出网接口**推断与公网通信的大致链路。Windows 优先 `Get-NetRoute` 取接口别名，回退 `route print`；Linux/macOS 用 `ip route get 8.8.8.8`（mac 回退 `route -n get`），再按接口名/别名关键词归类为 `wifi` / `cellular` / `ethernet`，无法识别时为 `unknown`（含 VPN/TUN 等保守处理）。
- **信号强度 `rssi_dbm`**：`ConnectionQualityCollector` 在无线场景下读取 dBm 或百分比再换算。Windows：`netsh wlan show interfaces`；Linux：优先 `iw dev <iface> link`，否则活跃 Wi‑Fi 的 `nmcli device show`；蜂窝可尝试 `mmcli -m 0`。无数据 或 采集失败时记为 **dmb的最小取值**（与「无读数」同形，前端按阈值着色时需结合 `network_type` 理解）。
- **丢包率 `packet_loss_rate`**：对公网地址 **`8.8.8.8` 与 `1.1.1.1` 顺序尝试 ICMP ping**（Windows `-n 8`，Linux/mac `-c 8`），解析控制台输出中的百分比或「已发/已收」推算，结果归一化到 **`0.0–1.0`**；两目标均失败时视为 **100% 丢包 `1.0`**。
- **延迟 `latency_ms`**：同上 ping 会话，优先从逐行 RTT 样本取**均值**；若无样本则从摘要行解析 **avg**（多语言 `ping`/`netsh` 输出兼容）；毫秒取整；读取失败时，赋值为`999ms`。
- **网络波动 `jitter_ms`**：有 ≥2 个 RTT 样本时用样本**总体标准差**；否则尝试从摘要中的 **mdev/stddev** 或 Windows/多语言「最小/最大/平均」估算半极差；毫秒取整;读取失败时，赋值为`999ms`。
- **公网 IP 路径 `IP_address`**：`IpCollector` 构造有序 IPv4 列表（去重保序）：**公网出口**（请求 `api.ipify.org` / `api64.ipify.org` JSON）、**默认网关**（Linux `ip route`/`route -n`，Windows `ipconfig`/`route print`）、**本机全局 IPv4**（`hostname -I` / `ip -4 addr` / `ipconfig` / UDP 探针回退），用于近似「经 NAT 到达观测侧的地址链」，**不是完整 traceroute**;读取失败时，赋值为`空列表`。

### `hardware_sender->backend_receiver` 传输阶段的数据结构：
```json
  "vita_type": "heartbeat",
  "vita_data": {
    "status": "online",        // 插件侧固定为 online；
    "network_type": "wifi",    // 访问公网的网络连接方式, wifi/蜂窝/有线
    "packet_loss_rate": 0.03,  // 丢包率
    "latency_ms": 36,          // 延迟
    "rssi_dbm": -45,           // 信号强度
    "jitter_ms": 10,           // 网络波动 
    "IP_address": ["203.0.113.8", "192.168.1.1", "10.0.0.12"]   // 公网出口 → 默认网关 → 本机 IPv4（见采集说明）
  }
```

### `backend_receiver`数据处理

暂无处理，赋予每个数据字段 对应的体征含义后，直接转发给前端。


### `backend_receiver->frontend_dashboard` 传输阶段的数据结构：
```json
  "vita_type": "heartbeat",
  "vita_data": {
    "network_type": "wifi",    // 访问公网的网络连接方式
    "packet_loss_rate": 0.03,  // 丢包率
    "latency_ms": 36,          // 延迟
    "rssi_dbm": -45,           // 信号强度
    "jitter_ms": 10,           // 网络波动 
    "IP_address": ["10.0.0.12", "203.0.113.8"]   // 该硬件访问到公网走的ip路线
  }
  "vita_description": {
    "network_type": "heartbeat_mode",    // 心跳模式
    "packet_loss_rate": "heartbeat_loss",// 心跳漏拍
    "latency_ms": "heartbeat_jam",       // 心跳阻塞
    "rssi_dbm": "heartbeat_force",       // 心跳强度
    "jitter_ms": "heartbeat_jitter",     // 心律不齐 
    "IP_address": "heartbeat_route"     // 血管路线
  }
```

### `frontend_dashboard`前端UI

设备详情页（`DeviceDetailView`）对 heartbeat 采用 **「最新值 + 近时序图」** 两层呈现：

- **最新一条**：与其它体征相同，经 `VitalColumn` 展示 `detail.latest.heartbeat` 各字段；字段名后的圆点颜色由 `trafficForHeartbeatField` 按阈值映射为绿/黄/红（`network_type`、`IP_address` 为中性灰）。阈值约定：**RSSI** ≥-55 绿、≥-70 黄、更弱为红；**延迟** ≤40ms 绿、≤120ms 黄、更高为红；**丢包率** 小于 1% 绿、小于 5% 黄、否则红；**抖动** ≤12ms 绿、≤35ms 黄、否则红。
- **近期心跳图**：`HeartbeatRecentChart` 使用 **ECharts** 折线图，在同一时间轴上叠加 **延迟 (ms)、信号强度 (dBm)、丢包率 (%)、抖动 (ms)** 四条曲线；RSSI 纵轴固定约 **-95～0 dBm** 以便 Wi‑Fi 场景对比。标题区说明为 **近 3 分钟** 窗口（`heartbeatSeries` 中 `WINDOW_MS` 与裁剪逻辑）。
- **数据来源与刷新**：进入页面拉取历史 `fetchVitalHistory(..., "heartbeat", 400)` 初始化序列；**每 8 秒** 轮询补全；同时 WebSocket `vital.updated` 实时合并新点（`server_time` 为秒级时间戳，图表内转为毫秒）。无数据时卡片内显示空状态提示。
