# 04 开发新体征插件

本文说明在**已有通信框架**下，如何为新的 `vita_type`（例如 `temperature`、`battery`）增加端到端支持。实现顺序建议：**先约定 JSON → 再写接收端插件（含推送前端）→ 再写发送端插件 → 最后补前端展示**（前端部分以仓库内 `frontend_dashboard` 实际代码为准）。

## 1. 设计阶段：固定四件事

1. **`vita_type` 字符串**  
   全局唯一、与 Topic 片段一致，例如 `temperature`。

2. **发送端 `vita_data` 结构**  
   硬件插件 `collect_real_vita_data()` / `collect_mock_vita_data()` 返回的 dict 即 MQTT 里 `vita_data` 的内容。字段名、类型、单位要写清楚，并与后端 `sender_vita_data_schema` 对齐。**两种模式产出同形载荷**。

3. **采集模式（`data_mode`）**  
   每个发送端插件须同时实现真实采集与模拟数据（见 [05 插件采集模式](05_插件采集模式_真实与模拟.md)）。`config.yaml` 用 `data_mode: real | mock` 切换；`kwargs` 仅服务真实采集，`mock` 段仅服务模拟。

4. **前端展示用 `vita_data`（可选映射）**  
   若与发送端完全一致，后端 `process_vita_data` 可做少量校验后直接返回；若需要单位换算、脱敏或补充 UI 专用字段，在 `BaseBackendVitalPlugin.process_vita_data` 中完成。

## 2. 后端插件：`backend_receiver`

### 2.1 放置位置

在 `backend_receiver/plugins/` 下新建包，例如 `plugins/temperature/`，其中包含 `plugin.py`（与现有 `heartbeat` 同级结构一致）。

### 2.2 类契约（继承 `BaseBackendVitalPlugin`）

必须提供：

| 项目 | 说明 |
| --- | --- |
| 类属性 `vita_type` | 与配置键、MQTT JSON 中一致。 |
| `sender_vita_data_schema` | 描述硬件侧字段，便于团队对齐。 |
| `frontend_vita_data_schema` | 描述推给前端的字段。 |
| `process_vita_data(sender_vita_data, envelope)` | 输入发送端载荷与完整 Envelope，返回 dict，作为前端 `vita_data`。 |
| `launch()` / `shutdown()` | 启动与停止插件内工作线程（或进程）；可参考 `HeartbeatBackendPlugin` 用线程循环调用 `process_once()`。 |

推送：由基类 `process_once` 编排——`process_vita_data` 后在**处理阶段** `history_infra.append_vital_event`（若 `persist_vital_history`），再 `publish_vital_update` 推送**本次** `frontend_vita_data`（不先查库）。插件内不得直接写 ClickHouse。

若该体征不宜自动落库（如大块 `local_map` 栅格），在 `kwargs` 中设 `persist_vital_history: false`，详见 `docs/04_History_Infra_ClickHouse.md`。

### 2.3 注册配置

在 `backend_receiver/config.yaml` 的 `receiver.plugins` 中增加一项，**键名等于 `vita_type`**，例如：

```yaml
receiver:
  plugins:
    temperature:
      class: "plugins.temperature.plugin:TemperatureBackendPlugin"
      enabled: true
      kwargs: {}
```

后端在 MQTT 连接成功后会**固定订阅** `robot/+/vitals/#`（以及可选的 `robot/+/status`）。`main` 里对每个已启用的 `vita_type` 调用 `mqtt.sub(vita_type, callback)` 的含义是：**注册该体征的处理回调**；若某条消息的 Envelope 中 `vita_type` 未注册回调，会在 MQTT Infra 层丢弃（debug 日志）。因此不必为每个新体征单独改订阅 Topic，但必须在后端 `receiver.plugins` 里注册，否则数据进不了业务插件。

## 3. 发送端插件：`hardware_sender`

### 3.1 放置位置

在 `hardware_sender/plugins/` 下新建目录，例如 `plugins/temperature/plugin.py`。

### 3.2 类契约（继承 `BaseVitalPlugin`）

必须实现：

| 项目 | 说明 |
| --- | --- |
| 类属性 `vita_type` | 与 YAML 键名一致。 |
| `vita_data_schema` | 文档化用途，描述真实/Mock 两种模式共同的 `vita_data` 结构。 |
| `collect_real_vita_data()` | 真实采集，返回 `dict`；访问硬件、ROS、HTTP 等。 |
| `collect_mock_vita_data()` | 模拟采集，返回 **同 schema** 的 `dict`；不访问外部依赖（可读本地 fixture）。 |

基类 `collect_vita_data()` 按 `data_mode` 分发，**子类不得覆盖**。采集循环仍由基类定时调用并 `put_data_to_pool`；**`put_data_to_pool` 负责在数据池 item 顶层写入 `data_mode`**，再经 MQTT Envelope 与后端 WS 事件全链路透传（见 [05 §5](05_插件采集模式_真实与模拟.md)）。

`collect_*` 与 `vita_data` 内**不得**手写 `data_mode`。

推荐将 Mock 逻辑放在 `plugins/<vita_type>/mock_generator.py`。

不要在该类中直接调用 MQTT：发布由主进程的 `MqttSenderInfra` 统一完成。

### 3.3 注册配置

在 `hardware_sender/config.yaml` 的 `sender.plugins` 中增加：

```yaml
sender:
  default_data_mode: real
  plugins:
    temperature:
      class: "plugins.temperature.plugin:TemperatureVitalPlugin"
      enabled: true
      data_mode: mock          # real | mock；开发联调可先用 mock
      interval_sec: 5.0
      qos: 0
      kwargs:
        sensor_i2c_bus: 1      # 仅 data_mode: real 时使用
      mock:
        seed: 0
        celsius_range: [20.0, 28.0]
```

注意：`sender.plugins` 的**键**必须与类上的 `vita_type` 完全相同，否则 `build_enabled_plugins` 会抛出 `ValueError`。`data_mode` 与 `mock` 约定见 [05 插件采集模式](05_插件采集模式_真实与模拟.md)。

## 4. 前端（`frontend_dashboard`）

架构上每种体征对应独立展示组件，消费 WebSocket 事件中自己的 `vita_type` 与 `vita_data`。具体文件路径与状态管理以 `docs/03_Frontend_前端展示.md` 及前
端源码为准；完成后需保证事件里的字段与 UI 绑定一致。

## 4. 内网自监控 Panel（`hardware_sender/local_dashboard/ui`）

若启用 **Local Dashboard**，在 `local_dashboard/ui/panels/` 增加 Python Panel 类，根据硬件 `vita_data` 渲染 Jinja 模板；在 `local_dashboard.vitals.{vita_type}` 配置 `ui_history_limit`（默认 20）。详见 `docs/05_Hardware_Local_Dashboard_内网自监控.md` §5.2。

云端大屏仍按 `docs/03_Frontend_前端展示.md` 在 `frontend_dashboard` 中实现。

## 5. 联调检查清单

- [ ] 发送端与接收端 `config.yaml` 中 `mqtt` 指向同一 Broker。  
- [ ] 两端 `vita_type` 字符串完全一致。  
- [ ] 接收端 `receiver.allowed_devices` 包含该设备 `device_id`（若使用白名单）。  
- [ ] Broker ACL 允许该账号对 `robot/{device_id}/vitals/{vita_type}` 发布与订阅。  
- [ ] 后端日志无「no plugin for vital」或队列满告警。  
- [ ] WebSocket 监听器能收到 `vita_type` 为新类型的 `vital.updated`（或当前实现中的等价事件名）。
- [ ] （若启用 Local Dashboard）内网页实时展示该体征；短历史条数 ≤ 配置的 `ui_history_limit`（默认 20）
- [ ] `data_mode: mock` 时无需真实硬件仍可全链路打通；`data_mode: real` 与现场行为一致。
- [ ] Mock 与 Real 的 `vita_data` 业务字段、类型、单位一致。
- [ ] 数据池 / MQTT / WebSocket 事件顶层均含 `data_mode`，且与 `config.yaml` 配置一致。
