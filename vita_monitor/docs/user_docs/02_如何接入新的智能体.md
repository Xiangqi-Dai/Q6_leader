# 02 如何接入新的智能体

「新增一台被监控的智能体」本质是：**为它分配全局唯一的 `device_id`，并在发送端与接收端配置中同时承认该身份**，且能连上同一个 MQTT Broker。

## 1. 前置条件

- 已可用的 MQTT Broker（地址、端口、TLS、用户名密码等）。
- 在 Broker 侧为该设备或设备组配置好 ACL（谁能发布/订阅哪些 Topic），与项目 readme 中 EMQX 账号流程一致。
- 接收端机器可访问公网 Broker；智能体侧网络可访问同一 Broker。

## 2. 在智能体上：配置 `hardware_sender`

1. 将 `hardware_sender/` 拷贝到设备（或镜像内安装依赖后挂载代码）。
2. 编辑 `hardware_sender/config.yaml`（或通过 `--config` 指定路径）：
   - **`device.id`**：为新智能体起一个**稳定且唯一**的字符串，例如 `robot_lab_03`。后续 Topic、后端 Worker、白名单都依赖它。
   - **`mqtt`**：`host`、`port`、`use_tls`、`username`、`password` 等与 Broker 及账号策略一致。
   - **`sender.plugins`**：启用需要的体征插件（如 `heartbeat`），`class` 使用 `模块路径:类名` 形式，`interval_sec`、`qos` 按需填写。

3. 启动发送端（在仓库根目录下示例；请按你本机 Python 环境调整）：

```bash
cd hardware_sender
python main.py
# 可选：python main.py --config /绝对路径/config.yaml
```

发送端启动后会连接 Broker、发布在线状态，并按插件周期上报体征。

## 3. 在监控服务器上：配置 `backend_receiver`

1. 编辑 `backend_receiver/config.yaml`：
   - **`mqtt`**：与发送端连接**同一套** Broker 参数（主机名、端口、认证）。
   - **`receiver.allowed_devices`**：在列表中加入新设备的 `device_id`。

     若列表**非空**，则只有列表内的 `device_id` 会被处理，其余设备的消息会被丢弃并打日志。列表为空时的行为以实现为准；生产环境建议**显式维护白名单**。

   - **`receiver.plugins`**：为每一种要在本机处理的 `vita_type` 配置后端插件类路径；发送端上报的类型必须在此有对应插件，否则消息会在设备 Worker 处因「无插件」被丢弃。

2. 启动后端：

```bash
cd backend_receiver
python main.py
# 可选：python main.py --config /绝对路径/config.yaml
```

3. 确认日志中出现 MQTT 连接成功、已订阅插件对应的 `vita_type`，并在收到数据后看到设备 Worker 或插件处理日志。

## 4. 在前端侧

- 打开 `frontend_dashboard`，连接后端提供的 **HTTP 基址** 与 **WebSocket 路径**（与后端 `frontend` 段配置一致）。
- 新设备首包到达后，后端通常会为其创建 Worker；前端应能通过 WebSocket 收到带新 `device_id` 的事件（具体 UI 是否自动出卡片取决于前端实现进度，见 `docs/03_Frontend_前端展示.md`）。

## 5. 常见问题

| 现象 | 可能原因 |
| --- | --- |
| 后端始终收不到数据 | Broker 地址/端口/防火墙不一致；用户名密码错误；发送端未启动。 |
| 日志提示 disallowed device | `receiver.allowed_devices` 未包含该 `device_id`。 |
| 提示 no plugin for vital | 后端 `receiver.plugins` 未注册该 `vita_type`，或键名与插件类上的 `vita_type` 不一致。 |
| 发送端无法连接 | 当前发送端 Infra 禁止将 `mqtt.host` 设为 localhost；远程设备应填公网或内网可达的 Broker 主机名。 |

## 6. 安全提醒

- **不要把**含生产密码的 `config.yaml` 提交到 Git 或复制到不可信环境。readme 中已有相关警告，部署时建议使用私密渠道分发配置。
